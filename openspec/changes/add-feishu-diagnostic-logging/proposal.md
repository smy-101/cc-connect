## Why

### 背景

当前仓库已经具备真实飞书长连接与文本发送能力，但在真实联调中，一旦出现“长连接 ready、主动发送成功、单聊消息无响应”的情况，现有日志不足以判断问题停留在飞书事件未下发、事件转换失败、消息路由失败，还是 Claude Code 调用失败。与此同时，`config.toml` 中的 `log_level` 目前没有真正控制 CLI 日志级别，导致调试阶段无法通过提升日志级别快速观察主链路状态。

### 目标

本次变更属于阶段 2 飞书平台适配器的联调收口工作，目标是在不改变现有消息与代理契约的前提下，为飞书入站、应用路由、回复发送和 Claude Code 调用补齐最小但可定位问题的诊断日志，并让 `log_level` 配置真正生效，降低真实环境排障成本。

## What Changes

### 范围

- 接通 `config.toml` 中的 `log_level`，使 CLI 能按 `debug`、`info`、`warn`、`error` 输出日志。
- 在 `internal/platform/feishu` 中增加飞书长连接、SDK 事件接收、事件转换、消息发送的结构化诊断日志。
- 在 `internal/app` 中增加文本消息处理、状态回复、Claude Code 调用和最终回复发送的关键断点日志。
- 明确记录会影响联调判断的核心字段，例如 `event_id`、`message_id`、`chat_type`、`message_type`、`channel_id`、回复阶段和错误原因。
- 保持默认日志级别下输出克制，仅在提升为 `debug` 时暴露详细诊断信息，避免正常运行时噪音过高。
- 补充对应测试，验证日志级别解析和关键错误路径不会因新增日志而改变原有行为。

### 非目标

- 不引入新的聊天平台或新的 AI 代理。
- 不改变统一消息模型、路由器、会话管理器和 Claude Code Agent 的对外契约。
- 不在本次变更中新增指标系统、持久化日志、诊断命令或 TUI 日志面板。
- 不把所有内部状态都暴露为日志，只覆盖真实联调所需的最小关键路径。

## Capabilities

### New Capabilities
- `diagnostic-logging`: 为飞书联调主链路提供可配置、结构化、可定位故障的运行时诊断日志，覆盖连接、事件接收、路由、回复发送与代理调用。

### Modified Capabilities
- 无

## Impact

### 影响模块

- `cmd/cc-connect`: 日志初始化与 `log_level` 配置接线。
- `internal/platform/feishu`: 长连接生命周期、SDK 事件入口、事件转换与消息发送诊断。
- `internal/app`: 文本消息处理、状态回复、Claude Code 调用与最终回复发送诊断。
- 测试：补充日志级别与关键错误路径测试，确保新增日志不改变既有行为。

### 验收标准

- 当 `config.toml` 设置 `log_level = "debug"` 时，CLI 能输出 debug 级别日志；设置为 `info` 时不输出 debug 日志。
- 飞书长连接 ready 后，收到真实单聊文本事件时，日志中能够依次显示事件到达、事件转换、路由处理和回复发送的关键阶段。
- 当飞书事件未能转换为统一消息时，日志中能够记录失败阶段与原因，而不是静默失败。
- 当回复发送失败时，日志中能够记录发送阶段、目标会话标识和错误原因。
- 当 Claude Code 调用失败或超时时，日志中能够记录调用阶段和失败结果，便于与飞书入站问题区分。
- 新增日志不会改变现有成功路径、错误返回语义和默认测试结果。

### 风险与缓解

- 日志过多导致正常运行噪音上升：通过 `debug` 级别承载详细链路日志，`info` 只保留关键状态。
- 结构化日志泄漏敏感信息：仅记录联调所需元数据，不打印 `app_secret`、完整用户输入或敏感凭证。
- 多层重复打印同一错误造成排障混乱：在连接层、适配层、应用层划分清晰的日志职责，避免同一错误被无差别重复记录。
