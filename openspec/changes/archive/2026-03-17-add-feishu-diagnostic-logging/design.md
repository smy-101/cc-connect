## Context

当前真实联调已经证明飞书出站消息链路可用，但入站消息无响应时，仓库缺少足以定位故障断点的结构化日志。现状存在两个直接问题：一是 `cmd/cc-connect` 没有把 `config.toml` 中的 `log_level` 接入实际 logger 初始化，导致调试期无法通过切换到 `debug` 放大观察面；二是飞书客户端、适配器和应用处理器在关键阶段缺少清晰的分层日志，一旦真实事件未落入主链路，很难区分是事件未下发、转换失败、路由失败，还是 Claude Code 调用失败。

本次设计属于阶段 2 飞书平台适配器的联调增强，不改变消息模型、路由契约和 AI 代理对外行为，只补充最小、结构化、可测试的诊断日志能力。

## Goals / Non-Goals

**Goals:**

- 让 `config.toml` 中的 `log_level` 真正控制 CLI 日志输出级别。
- 为飞书入站消息主链路增加分层诊断日志，覆盖连接、事件接收、事件转换、路由、状态回复、代理调用和最终回复发送。
- 通过结构化字段输出关键上下文，支持根据日志快速判断故障停留在哪一层。
- 保持默认 `info` 级别输出简洁，仅在 `debug` 级别暴露详细链路日志。
- 以测试保护新增行为，避免日志增强改变既有成功路径与错误语义。

**Non-Goals:**

- 不引入新的日志框架、分布式 tracing、metrics 或持久化日志存储。
- 不新增诊断 CLI 命令、TUI 日志面板或后台管理页面。
- 不把全部业务数据写入日志，尤其不打印凭证、完整原始消息内容或其它敏感字段。

## Decisions

### D1: 继续使用 `log/slog`，由配置驱动日志级别

**选择**：保留当前 `log/slog` 作为唯一日志实现，在 `loadConfig` 成功后根据 `config.LogLevel` 重建默认 logger，使 `debug`、`info`、`warn`、`error` 与配置一致。

**理由**：

- 仓库已经使用 `slog`，继续沿用可以把改动限制在初始化与调用点，不引入新的依赖和迁移成本。
- `slog` 原生支持结构化字段，适合表达 `event_id`、`message_id`、`chat_type` 这类联调上下文。

**替代方案**：

- 引入第三方日志库：功能更丰富，但对当前“最小诊断增强”收益不足。
- 继续固定 `info` 级别：实现最小，但无法满足联调排障需求。

### D2: 按层定义日志职责，避免重复与吞错

**选择**：把日志职责分为三层。

- `internal/platform/feishu/client_impl.go` 与 `sdk_facade.go`：记录连接生命周期、SDK 事件入口、发送 API 结果。
- `internal/platform/feishu/adapter.go`：记录事件转换、命令识别与路由结果。
- `internal/app/handlers.go` 与 `reply.go`：记录状态回复、Claude Code 调用和最终回复发送。

**理由**：

- 故障定位依赖知道“哪一层看到过这条消息”。如果所有层都无差别打印，会制造噪音；如果只有最外层打印，一旦中间层吞错就难以定位。
- 分层职责能确保每个错误只在最贴近语义的位置记录一次高价值日志。

**替代方案**：

- 只在最外层统一打印：实现简单，但无法区分事件未下发与中间转换失败。
- 每层全面打印所有内容：信息过载，且容易重复记录同一错误。

### D3: `debug` 记录链路阶段，`info` 仅记录关键状态

**选择**：

- `info`：保留启动、长连接 ready、关闭、关键失败等用户可感知状态。
- `debug`：记录收到 SDK 事件、事件转换成功、进入 handler、发送“正在思考”、调用 agent、发送最终回复等链路细节。
- `warn` / `error`：记录异常、失败与会影响联调结果的问题。

**理由**：

- 正常运行时不应被联调细节日志淹没。
- 调试时又必须能完整观察主链路。

**替代方案**：

- 所有日志都放在 `info`：能看到更多细节，但会让默认运行噪音过高。
- 只有错误日志：当消息完全丢失时，无法区分“没收到”和“没处理”。

### D4: 关键字段以元数据为主，避免敏感内容泄漏

**选择**：日志中优先记录以下元数据：`event_id`、`message_id`、`chat_id` 或 `channel_id`、`chat_type`、`message_type`、`sender_type`、阶段名称、错误原因、内容长度。默认不打印 `app_secret`、完整凭证、完整用户输入和完整回复文本。

**理由**：

- 联调定位通常不需要完整消息内容，元数据足以判断链路阶段。
- 减少把用户消息、凭证等敏感信息写入终端或 CI 日志的风险。

**替代方案**：

- 打印完整消息内容：调试更直接，但风险高，不适合作为默认实现。

### D5: 用测试验证“能看见日志”与“行为不回归”

**选择**：测试重点放在两类场景。

- 日志级别解析与初始化：验证 `debug` 能启用细节日志，`info` 不输出 debug 日志。
- 关键错误路径：验证事件转换失败、回复发送失败、agent 调用失败等路径仍保留原错误语义，并且可被日志覆盖。

**理由**：

- 这次变更的核心价值是可观测性，至少要证明日志级别接线有效。
- 同时不能因为加日志改变现有错误返回或测试行为。

**替代方案**：

- 完全不测日志：速度快，但容易出现“代码改了、调试仍无效”的假完成状态。

## Risks / Trade-offs

- [`main.go` 在加载配置前无法使用配置级别日志] → 采用两阶段初始化：先用保守默认 logger 读取配置，加载成功后按配置重建 logger。
- [飞书链路多处打日志造成重复] → 明确每层只记录该层最有语义价值的状态与错误，避免重复打印同一错误堆叠。
- [日志字段不足仍难定位问题] → 统一约定关键字段集合，并在设计中要求覆盖 `event_id`、`message_id`、`chat_type`、`message_type`、阶段名。
- [调试信息泄漏敏感数据] → 默认不记录凭证与完整消息正文，必要时仅记录长度或标识符。

## Migration Plan

1. 在 CLI 启动流程中接通 `config.LogLevel`，提供稳定的日志级别解析函数。
2. 在飞书客户端、适配器、应用处理器和发送路径补齐结构化日志，并确保失败路径不再静默。
3. 为日志级别接线和关键错误路径补充测试，验证默认测试不回归。
4. 使用真实联调场景复测：运行 `debug` 级别，发送单聊文本，确认日志足以判断链路断点。

**回滚策略：**

- 若新增日志干扰现有行为，可先回退日志调用点，保留 `log_level` 接线。
- 若日志量不可接受，可把部分链路日志从 `info` 下调为 `debug`，而不改变功能行为。

## Open Questions

- 是否需要为后续 TUI 日志面板保留统一的日志字段命名约定；若当前不需要，可先只保证 CLI 输出一致。
- `debug` 级别下是否需要选择性打印部分短文本摘要；当前建议先不做，只保留元数据与长度。
