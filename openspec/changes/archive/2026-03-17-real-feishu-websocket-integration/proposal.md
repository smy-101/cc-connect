## Why

### 背景

当前仓库已经声明支持飞书 WebSocket 长连接，但实际实现仍停留在占位状态：`internal/platform/feishu/client_impl.go` 仅模拟连接成功，`go.mod` 也未引入飞书官方 Go SDK。结果是应用可以在本地打印“启动成功”，却无法真正建立与飞书开放平台的长连接，也无法完成飞书后台要求的“先启动长连接、再继续事件配置”的接入流程。

### 目标

本次变更属于阶段 2 飞书平台适配器的兑现和收口工作，目标是将历史设计中的真实长连接能力落地到当前仓库，并补齐事件配置指引与真实联调验收，解除 `mvp-integration` 中真实环境手测项的阻塞。

## What Changes

### 范围

- 以飞书官方 Go SDK 替换当前模拟长连接实现，建立真实 WebSocket 长连接。
- 保持现有 `FeishuClient` 抽象，对外行为不变，补齐真实连接状态、断开、重连和事件回调处理。
- 将发送文本消息的实现从占位逻辑替换为真实飞书 API 调用路径。
- 增加启动期连接就绪反馈和失败诊断，避免“本地看似已连接、实际未接入飞书平台”的假阳性状态。
- 更新 README 和接入说明，明确“先启动 cc-connect 建立长连接，再到飞书后台继续事件配置”的操作顺序。
- 增加真实联调验收清单，覆盖启动、事件配置、消息接收、消息回复、断开与恢复等关键场景。

### 非目标

- 不扩展新的聊天平台。
- 不引入飞书卡片消息、图片下载、语音处理等后续阶段能力。
- 不改变统一消息模型、消息路由、会话管理和 Claude Code 代理的既有对外契约。

## Capabilities

### New Capabilities

- `feishu-websocket-client`: 真实飞书 WebSocket 长连接客户端，覆盖官方 SDK 接入、连接建立、连接就绪反馈、事件接收、发送文本消息、断线恢复与可观测性。

### Modified Capabilities

- 无

## Impact

### 影响模块

- `internal/platform/feishu`: 真实 SDK 客户端、发送器、事件处理链路。
- `internal/app`: 应用启动和关闭流程中的飞书连接建立、就绪诊断和错误传播。
- `cmd/cc-connect`: 启动日志和联调时的用户可见反馈。
- `go.mod` / `go.sum`: 引入飞书官方 Go SDK 及其依赖。
- `README.md` 与相关文档：补充事件订阅前置步骤和联调指引。
- 测试：补充单元测试、mock 契约测试、集成测试及真实环境验收记录。

### 验收标准

- 使用有效 `app_id` / `app_secret` 启动后，应用能够建立真实飞书长连接，而不是仅在内存中标记为已连接。
- 长连接建立成功后，应用输出明确的“连接就绪”反馈，足以指导操作者继续飞书后台事件配置。
- 使用无效凭证或连接失败时，应用启动失败并返回明确错误，不得出现假阳性连接状态。
- 在飞书后台完成事件配置后，真实 `im.message.receive_v1` 事件能够到达应用并进入现有消息路由链路。
- 应用能够通过真实飞书 API 发送文本回复消息。
- README 中提供可执行的接入顺序说明，覆盖编译、启动、飞书后台配置和验证步骤。
- 真实联调验收至少覆盖：连接建立、事件配置完成、文本消息收发、断开后恢复。

### 风险与缓解

- SDK API 与当前占位接口存在偏差：通过保留 `FeishuClient` 接口并在实现层新增 SDK 封装，控制改动面。
- 真实长连接引入外部网络不稳定因素：通过超时、重试、断线恢复和明确日志降低排障成本。
- 真实联调依赖飞书凭证和外部环境：通过 mock 测试与 build tag 集成测试分层验证，真实验收仅覆盖关键主链路。
- 文档若仍与实现脱节会再次误导用户：把 README 接入顺序和真实验收清单纳入同一 change 管理和验收。
