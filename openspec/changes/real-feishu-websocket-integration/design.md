## Context

当前 `internal/platform/feishu/client_impl.go` 仍是占位实现：`Connect` 只校验凭证是否为空并把内部状态置为已连接，没有真正与飞书开放平台建立 WebSocket 长连接，也没有接入真实事件回调和消息发送路径。这与仓库既有设计、README 对外承诺以及 `mvp-integration` 中“真实配置手测”的目标不一致。

本次变更是一次跨模块收口：需要同时调整 `internal/platform/feishu` 的真实 SDK 接入、`internal/app` 的启动错误传播和连接就绪反馈、README 的接入顺序，以及真实联调验收方式。约束是保持现有核心域模型和 `FeishuClient` 抽象稳定，避免把 SDK 细节泄漏到 `core` 或 `app`。

## Goals / Non-Goals

**Goals:**

- 在不改变现有核心路由和会话契约的前提下，引入真实飞书官方 Go SDK，建立可工作的长连接。
- 明确区分“已启动进程”和“已建立飞书长连接”两个状态，提供可操作的连接就绪反馈。
- 让真实 `im.message.receive_v1` 事件进入既有转换和路由链路，并能通过真实 API 发送文本回复。
- 为测试保留 mock 边界，在默认单测中不依赖真实网络，同时补充可选的真实集成验证。
- 更新 README 和验收清单，补齐“先启动长连接，再去飞书后台继续事件配置”的联调流程。

**Non-Goals:**

- 不扩展卡片消息、图片下载、语音转写等阶段外功能。
- 不改变 `core.Message`、`core.Router`、`command.Executor` 或 Claude Code Agent 的对外接口。
- 不在本次变更中处理多实例部署、消息分片或高可用集群问题。

## Decisions

### D1: 使用飞书官方 Go SDK 的 WebSocket 与消息发送能力

**选择**：在 `internal/platform/feishu` 内引入飞书官方 Go SDK，使用其长连接能力建立 WebSocket 连接，并使用官方消息发送 API 完成回复。

**理由**：

- 官方 SDK 已覆盖鉴权、心跳、重连等关键行为，能减少自实现协议的错误面。
- 这与历史飞书适配器设计保持一致，能把现有占位实现补成真实能力，而不是引入第二套架构。

**替代方案**：

- 自行实现 WebSocket 协议：灵活但工作量大，且会重复踩飞书协议与重连细节。
- 第三方封装库：集成成本可能更低，但维护质量和升级节奏不可控。

### D2: 保留 `FeishuClient` 接口，在实现内部增加 SDK facade

**选择**：保持 `FeishuClient` 作为对上层的稳定边界，在 `client_impl.go` 内增加一层私有 facade，封装 SDK 客户端创建、事件注册、连接启动和消息发送。

**理由**：

- `app`、`sender`、测试桩都已经围绕 `FeishuClient` 组织，保留接口可以控制影响面。
- facade 允许在单测里模拟 SDK 回调、连接错误和重连行为，而不是强依赖真实 SDK 对象。

**替代方案**：

- 在各处直接依赖 SDK 对象：改动更少，但会让测试和后续升级成本明显变差。

### D3: 引入明确的连接状态机和“就绪”语义

**选择**：将连接状态拆分为 `disconnected`、`connecting`、`ready`、`stopping`，只有在 SDK 确认长连接真正建立后才把客户端标记为 ready；任何启动失败都必须让 `Connect` 返回错误并保持未连接状态。

**理由**：

- 当前最大问题不是单纯“连不上”，而是“看起来连上了”。
- 飞书后台事件配置依赖真实长连接存在，因此需要一个可以直接指导操作者的 ready 信号。

**替代方案**：

- 延续当前布尔状态：实现简单，但会继续制造假阳性，无法支撑真实联调。

### D4: 事件处理继续使用异步 worker，但 ACK 责任交给 SDK 回调约束

**选择**：SDK 事件回调只做最小解析和入队，业务处理继续交给现有 adapter/router 链路异步执行；错误通过日志和指标暴露，不阻塞连接主循环。

**理由**：

- 飞书事件处理存在时限约束，AI 处理链路又可能较慢，异步边界仍然必要。
- 这样可以最大化复用现有 `Adapter.HandleEvent` 与 message converter 逻辑。

**替代方案**：

- 在 SDK 回调内同步完成所有业务处理：实现直接，但容易造成回调超时和重推。

### D5: 将真实网络验证与默认测试分层

**选择**：默认测试继续使用 mock/facade；真实 SDK 验证使用 build tag 或环境变量门控的集成测试，并保留一份手工联调验收清单。

**理由**：

- 仓库的默认 CI 不应依赖真实飞书网络和凭证。
- 这次变更必须证明真实环境可用，因此仅有 mock 测试不够。

**替代方案**：

- 全部只做 mock：速度快，但无法证明长连接真的可用。
- 默认测试直接连真实飞书：不可重复，环境成本过高。

### D6: 文档作为联调能力的一部分交付

**选择**：README 和联调说明纳入同一 change，明确前置条件、启动顺序、飞书后台配置步骤、成功标志和排障入口。

**理由**：

- 当前用户遇到的问题本质上是“实现未兑现”和“接入路径不清晰”的叠加。
- 即使代码接通，如果文档仍然按旧顺序描述，集成依然会卡住。

## Risks / Trade-offs

- [SDK 实际 API 与预期存在偏差] → 通过 facade 隔离、先补契约测试、实现前用小范围原型验证构造函数和回调签名。
- [长连接启动语义可能是阻塞式] → 在 `Connect(ctx)` 中显式管理 goroutine、取消和关闭路径，避免 `App.Start()` 无法返回或提前返回。
- [断线恢复带来状态竞争] → 用互斥锁和单一状态机维护连接状态，重连逻辑只在实现层出现，不泄漏到 `app`。
- [真实联调依赖外部凭证] → 默认自动化测试仍使用 mock，真实环境仅做最小必要验证和手工记录。
- [文档与实现再次漂移] → 把 README 更新和真实联调验收列入 tasks，同步作为 apply 前后的验收项。

## Migration Plan

1. 引入飞书官方 Go SDK，并在 `internal/platform/feishu` 内建立 facade 与契约测试。
2. 用真实 SDK 替换当前占位 `Connect` / `Disconnect` / `SendText` 逻辑，补齐事件注册和状态管理。
3. 调整 `internal/app` 启动流程中的错误传播和就绪反馈，让启动日志能区分“进程启动成功”和“飞书连接就绪”。
4. 增加真实 SDK 集成测试入口与手工联调清单。
5. 更新 README 接入文档，明确启动与飞书后台配置顺序。

**回滚策略：**

- 若真实 SDK 接入导致严重不稳定，可回滚到本次 change 之前的提交。
- 临时停用飞书能力时，可在配置中关闭对应项目的 `projects.feishu.enabled`。

## Open Questions

- 官方 SDK 的长连接启动 API 是否天然阻塞到 `ctx` 取消，还是可以在连接建立后立即返回；这会影响 `Connect` 的实现形态。
- 消息发送是否应复用同一 SDK client，还是单独构造 API service client 更清晰；以实现复杂度更低且易测的方案为准。
- 是否需要在启动日志之外额外输出结构化状态摘要，供未来 TUI 或诊断命令复用；如超出本次最小范围，则仅先输出清晰日志。
