# 配置管理系统

## Why

当前 cc-connect 已实现统一消息结构、消息路由和会话管理的核心功能，但缺少统一的配置管理系统。后续的飞书适配器和 Claude Code 适配器都需要读取平台凭证、代理设置等配置信息。在进入阶段 2 之前，需要先建立配置加载、验证和管理的标准化机制。

## What Changes

- 新增 TOML 格式的配置文件支持
- 实现应用级和项目级的两层配置结构
- 支持环境变量引用语法 `${VAR}` 处理敏感信息
- 提供配置验证和启动时摘要报告功能
- 支持多项目配置管理（为后续阶段预留能力）

## Capabilities

### New Capabilities

- `config-management`: 配置管理能力，包括 TOML 文件解析、环境变量展开、配置验证、多项目支持和启动摘要报告

### Modified Capabilities

无。本变更不修改现有能力的规格要求。

## Impact

### 影响模块

| 模块 | 影响 |
|------|------|
| `internal/core` | 新增 `config.go`, `config_test.go` |
| `internal/core` | SessionConfig 可被项目级配置覆盖 |
| 第三方依赖 | 新增 `github.com/pelletier/go-toml/v2` |

### 影响能力域

- **core**: 新增配置管理核心模块
- **platform/feishu**: 后续将通过配置获取 App ID/Secret
- **agent/claudecode**: 后续将通过配置获取工作目录和权限模式

## 背景

本变更属于 **阶段 1: 核心消息系统** 的最后一部分。当前已完成：
- ✅ 统一消息结构 (unified-message)
- ✅ 消息路由引擎 (message-router)
- ✅ 会话管理系统 (session-management)
- ❌ 配置管理系统 ← 本变更

## 目标

1. 提供标准化的 TOML 配置文件加载能力
2. 支持环境变量引用，避免敏感信息硬编码
3. 支持多项目配置结构，每个项目可独立配置平台和代理
4. 启动时输出配置验证报告，提升用户体验
5. 为后续飞书和 Claude Code 适配器提供配置基础

## 范围

### 包含

- TOML 文件解析和反序列化
- 应用级配置（日志级别、默认项目）
- 项目级配置（名称、工作目录、平台配置、代理配置）
- 环境变量展开（`${VAR}` 语法）
- 配置验证（必填检查、格式检查、路径存在性检查）
- 配置摘要报告（启用状态、警告信息）
- 多项目配置结构（支持定义多个项目）

### 不包含

- 热重载（运行时配置更新）
- 配置文件自动生成向导
- 远程配置拉取
- 配置加密存储
- 实际的多项目运行时切换（仅建立数据结构）

## 非目标

- 不实现配置的 GUI 编辑功能（属于 TUI 阶段）
- 不实现配置变更的持久化监听
- 不实现多项目的并发运行管理

## 验收标准

### 功能验收

| 场景 | 预期结果 |
|------|----------|
| 加载有效 TOML 文件 | 成功解析为 AppConfig 结构 |
| 加载不存在文件 | 返回明确错误信息 |
| 加载格式错误 TOML | 返回解析错误 |
| 环境变量 `${VAR}` | 正确展开为环境变量值 |
| 缺失环境变量 | 空字符串 + 验证警告 |
| 必填字段缺失 | 返回验证错误 |
| 无效权限模式 | 返回验证错误 |
| 工作目录不存在 | 返回验证错误 |
| 多项目配置 | 正确解析所有项目 |
| 默认项目不存在 | 返回验证错误 |
| 项目名称重复 | 返回验证错误 |

### 质量验收

- 测试覆盖率 ≥ 85%
- 所有测试通过 `go test -race`
- 配置加载无外部依赖（文件 I/O 除外）

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| TOML 格式变更 | 用户配置失效 | 严格的版本兼容性，变更时提供迁移指南 |
| 敏感信息泄露 | 安全风险 | 环境变量支持 + 摘要报告脱敏显示 |
| 配置复杂度过高 | 用户难以上手 | 提供示例配置文件 + 清晰的错误提示 |
